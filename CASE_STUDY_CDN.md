## The Six-Phase Delivery Workflow

Seed AI recognized and navigated the industrial delivery lifecycle without being explicitly told which phase to enter:

### Phase 1 — Requirements Analysis

**User prompt:** "我需要从零构建一套CDN边界工业系统"

**Seed AI's response:** Immediately asked the right clarifying questions:
- Multi-region or single-node?
- What origin protocol (HTTP/S, WebSocket, gRPC)?
- TLS management: self-signed or ACME?
- Rate limiting: global or per-client?

This phase produced a requirements spec that was written to `CLAUDE.md` at project root, persisting across all subsequent sessions.

---

### Phase 2 — Architecture Design

Before writing any code, Seed AI produced a `docs/architecture.md` with:
- Component interaction diagram (ASCII)
- Data flow: client → edge → origin
- Technology selection rationale (OpenResty over plain Nginx: Lua scripting; Go over Python: static binary for Docker)
- Risk register (OpenResty module availability on Alpine, Docker DNS resolution in Compose networks)

**Key decision preserved in memory:**
> Chose cfssl over certbot because certbot requires ACME server access; cfssl generates offline cert chains suitable for air-gapped environments.

---

### Phase 3 — Environment Preparation

This phase exposed the most environment-specific challenges — all of which Seed AI diagnosed and resolved autonomously.

#### Problem 1: OpenResty missing `lua-resty-redis` module

```
nginx: [error] module "resty.redis" not found
```

**Root cause diagnosed by Seed AI:** The `openresty/openresty:alpine` Docker image ships without `opm` (OpenResty Package Manager) pre-installed. `lua-resty-redis` is not bundled by default.

**Fix generated:**
```dockerfile
FROM openresty/openresty:alpine
RUN opm get ledgetech/lua-resty-redis
```

#### Problem 2: Nginx DNS resolution failure inside Docker Compose

```
[error] 15#15: recv() failed (111: Connection refused) while resolving, resolver: 127.0.0.53
```

**Root cause:** Nginx resolves DNS at config load time using the host's `/etc/resolv.conf`. Inside Docker Compose, service names resolve via Docker's embedded DNS at `127.0.0.11`, not `127.0.0.53`.

**Fix generated by Seed AI:**
```nginx
resolver 127.0.0.11 valid=30s ipv6=off;
set $backend "origin-service";
proxy_pass http://$backend;
```

*Note: Using a variable forces runtime DNS resolution, bypassing Nginx's startup-time lookup.*

#### Problem 3: Docker Compose network naming

```
Error: network "cdn_internal" not found
```

**Root cause:** Docker Compose prefixes network names with the project directory name: `<dirname>_<network-name>`. A project in `/home/user/cdn-edge/` creates `cdn-edge_internal`, not `cdn_internal`.

**Fix:** Seed AI generated an explicit network name in `docker-compose.yml`:
```yaml
networks:
  internal:
    name: cdn_internal
```

### Phase 4 — Core Implementation

Seed AI wrote all core components. Notable patterns:

#### Lua rate limiter (`lua/rate_limit.lua`)
```lua
local redis = require "resty.redis"
local red = redis:new()
red:set_timeouts(100, 100, 100)

local ok, err = red:connect("redis", 6379)
if not ok then
    ngx.log(ngx.ERR, "Redis connect failed: ", err)
    return  -- fail open, not closed
end

local key = "rl:" .. ngx.var.remote_addr
local count, err = red:incr(key)
if count == 1 then
    red:expire(key, 60)
end

if count > 100 then
    ngx.exit(429)
end
```

**Design decision (from Seed AI):** "Fail open" on Redis unavailability — rate limiter unavailability should not cause edge node outage.

#### Go origin health checker (`cmd/healthcheck/main.go`)
```go
func main() {
    origins := strings.Split(os.Getenv("ORIGINS"), ",")
    ticker := time.NewTicker(10 * time.Second)
    for range ticker.C {
        for _, origin := range origins {
            go checkOrigin(origin)
        }
    }
}
```

Cross-compiled for Docker container:
```bash
GOOS=linux GOARCH=amd64 go build -o bin/healthcheck ./cmd/healthcheck
```

### Phase 5 — Integration Testing

Seed AI generated a complete test matrix and executed it via `bash` tool:

```bash
# TLS handshake test
curl -v --cacert certs/ca.pem https://localhost:8443/health

# Rate limit trigger test  
for i in $(seq 1 110); do
    curl -s -o /dev/null -w "%{http_code}" https://localhost:8443/
done | tr -d '\n'
# Expected: 100×200 followed by 10×429

# Origin failover test
docker compose stop origin-primary
sleep 15
curl -s https://localhost:8443/health | jq .status
# Expected: "degraded" with secondary origin active
```

All tests passed on first run after fixes in Phase 3.

## Problems That Improved Seed AI

The CDN project didn't just validate Seed AI — it *improved* it. Each friction point was analyzed, root-caused, and fixed:

| Problem Encountered | Root Cause | Fix Applied to Seed AI |
|---------------------|------------|------------------------|
| AI forgot instructions after 3-4 turns | `compactWithSummary` and `extractFromConversation` hardcoded Anthropic SDK — DeepSeek users always 403 | Provider-agnostic AIProvider abstraction for all internal LLM calls |
| System stopped after 50 tool calls | `MAX_ITERATIONS = 50` too low for complex multi-phase projects | Raised to 200 |
| Whitepaper truncated to "simplified version" | Global conciseness rules applied to document generation | Added explicit `# Document generation EXCEPTION` to system prompt |
| "Max tokens reached" hard stop | No auto-continue logic | Auto-continue with dummy tool_results + targeted file_write resume message |
| Terminal flickered during tool execution | Completed tool_use blocks dropped (N lines → 0), Ink cursor drift | Replace completed blocks with 1-line `✓ tool_name` (N → 1) |
| AI had no project context after restart | No `CLAUDE.md` in project root | `/init` command added to scaffold project context file |
| AI didn't know OpenResty/Docker/cfssl specifics | System prompt had no infra domain knowledge | CDN/container/certificate knowledge section added to system prompt |

## Lessons Learned

### What Worked Exceptionally Well

1. **Parallel tool execution (I001)** — reading multiple config files simultaneously, running tests in parallel batches — noticeably faster than sequential execution.

2. **Cross-session memory (I007)** — after fixing the Anthropic hardcode bug, Seed AI correctly recalled architectural decisions from session 1 in session 3: *"Based on our earlier decision to use cfssl for offline cert generation..."*

3. **`/diag` command (I017)** — instead of describing error symptoms manually, running `/diag` instantly surfaced the root cause from `~/.seed/debug.log`. Cut diagnosis time from minutes to seconds.

4. **`/init` + CLAUDE.md** — creating `CLAUDE.md` at session start meant Seed AI never asked "what is this project?" again, even after restarts and compactions.

5. **Document generation exception** — the whitepaper was written completely and correctly in one pass: section-by-section, no truncation, no "simplified version" offers.

### What Required Manual Intervention

1. **Model intelligence gap** — DeepSeek-chat occasionally needed explicit re-prompting for complex multi-constraint problems (e.g., Nginx variable trick for runtime DNS). Claude Sonnet 4.6 handles these without hints.

2. **First-run Docker DNS** — Seed AI diagnosed correctly but the fix required understanding Docker's embedded DNS architecture. The fix is now baked into the system prompt for future projects.

## Conclusion

The CDN edge project demonstrated that Seed AI can serve as a complete engineering partner for industrial-grade systems:

- **Architecture** to **implementation** to **documentation** in a single coherent workflow
- **Cross-session memory** that actually works — architectural decisions persist across restarts
